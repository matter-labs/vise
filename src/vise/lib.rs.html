1//! Metrics handling library based on the `prometheus-client` crate.
2//!
3//! # Overview
4//!
5//! - The crate supports defining common metric types ([`Counter`]s, [`Gauge`]s and [`Histogram`]s).
6//!   A single metric is represented by an instance of these types; it can be reported using methods
7//!   like [`Counter::inc()`], [`Gauge::set()`] or [`Histogram::observe()`].
8//! - Metrics can be grouped into a [`Family`]. Essentially, a `Family` is a map in which metrics
9//!   are values keyed by a set of labels. See [`EncodeLabelValue`] and [`EncodeLabelSet`] derive macros
10//!   for more info on labels.
11//! - To define metrics, a group of logically related metrics is grouped into a struct
12//!   and the [`Metrics`](trait@Metrics) trait is [derived](macro@Metrics) for it. This resolves
13//!   full metric names and records additional metadata, such as help (from doc comments), unit of measurement
14//!   and [`Buckets`] for histograms.
15//! - Metric groups are registered in a [`Registry`], which then allows to [encode](Registry::encode())
16//!   metric data in the OpenMetrics text format. Registration can be automated using the [`register`]
17//!   attribute, but it can be manual as well.
18//! - In order to allow for metrics computed during scraping, you can use [`Collector`].
19//! - To share one or more labels for a group of metrics, wrap them in a [`MetricsFamily`].
20//!
21//! # Examples
22//!
23//! ## Defining metrics
24//!
25//! ```
26//! use vise::*;
27//! use std::{fmt, time::Duration};
28//!
29//! /// Metrics defined by the library or application. A single app / lib can define
30//! /// multiple metric structs.
31//! #[derive(Debug, Clone, Metrics)]
32//! #[metrics(prefix = "my_app")]
33//! // ^ Prefix added to all field names to get the final metric name (e.g., `my_app_latencies`).
34//! pub(crate) struct MyMetrics {
35//!     /// Simple counter. Doc comments for the fields will be reported
36//!     /// as Prometheus metric descriptions.
37//!     pub counter: Counter,
38//!     /// Integer-valued gauge. Unit will be reported to Prometheus and will influence metric name
39//!     /// by adding the corresponding suffix to it (in this case, `_bytes`).
40//!     #[metrics(unit = Unit::Bytes)]
41//!     pub gauge: Gauge<u64>,
42//!     /// Group of histograms with the "method" label (see the definition below).
43//!     /// Each `Histogram` or `Family` of `Histogram`s must define buckets; in this case,
44//!     /// we use default buckets for latencies.
45//!     #[metrics(buckets = Buckets::LATENCIES)]
46//!     pub latencies: Family<Method, Histogram<Duration>>,
47//! }
48//!
49//! /// Isolated metric label. Note the `label` name specification below.
50//! #[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, EncodeLabelSet, EncodeLabelValue)]
51//! #[metrics(label = "method")]
52//! pub(crate) struct Method(pub &'static str);
53//!
54//! // For the isolated metric label to work, you should implement `Display` for it:
55//! impl fmt::Display for Method {
56//!     fn fmt(&self, formatter: &mut fmt::Formatter<'_>) -> fmt::Result {
57//!         write!(formatter, "{}", self.0)
58//!     }
59//! }
60//! ```
61//!
62//! ## Registering metrics automatically
63//!
64//! Commonly, metrics can be registered by defining a `static`:
65//!
66//! ```
67//! # use vise::{Gauge, Global, Metrics, MetricsCollection, Registry};
68//! #[derive(Debug, Clone, Metrics)]
69//! pub(crate) struct MyMetrics {
70//! #   pub gauge: Gauge<u64>,
71//!     // defined in a previous code sample
72//! }
73//!
74//! #[vise::register]
75//! pub(crate) static MY_METRICS: Global<MyMetrics> = Global::new();
76//!
77//! // All registered metrics can be collected in a `Registry`:
78//! let registry: Registry = MetricsCollection::default().collect();
79//! // Do something with the `registry`, e.g. create an exporter.
80//!
81//! fn metered_logic() {
82//!     // method logic...
83//!     MY_METRICS.gauge.set(42);
84//! }
85//! ```
86//!
87//! ## Registering metrics manually
88//!
89//! It is possible to add metrics manually to a [`Registry`]. As a downside, this approach requires
90//! boilerplate to register all necessary metrics in an app and potentially libraries
91//! that it depends on.
92//!
93//! ```
94//! # use vise::{Gauge, Metrics, Registry};
95//! #[derive(Debug, Clone, Metrics)]
96//! pub(crate) struct MyMetrics {
97//! #   pub gauge: Gauge<u64>,
98//!     // defined in a previous code sample
99//! }
100//!
101//! let mut registry = Registry::empty();
102//! let my_metrics = MyMetrics::default();
103//! registry.register_metrics(&my_metrics);
104//! // Do something with the `registry`, e.g. create an exporter.
105//!
106//! // After registration, metrics can be moved to logic that reports the metrics.
107//! // Note that metric types trivially implement `Clone` to allow sharing
108//! // them among multiple components.
109//! fn metered_logic(metrics: MyMetrics) {
110//!     // method logic...
111//!     metrics.gauge.set(42);
112//! }
113//!
114//! metered_logic(my_metrics);
115//! ```

162/// Derives the [`EncodeLabelSet`] trait for a type, which encodes a set of metric labels.
163///
164/// The type for which the trait is derived must not have generics (lifetimes, type params or const params).
165/// The macro can be configured using `#[metrics()]` attributes.
166///
167/// # Container attributes
168///
169/// ## `label`
170///
171/// **Type:** string
172///
173/// If specified, the type will be treated as a single label with the given name. This covers
174/// the common case in which a label set consists of a single label. In this case, the type
175/// also needs to implement [`EncodeLabelValue`].
176///
177/// If this attribute is not specified (which is the default), a container must be a `struct`
178/// with named fields. A label with the matching name will be created for each field.
179///
180/// # Field attributes
181///
182/// ## `skip`
183///
184/// **Type:** path to a function with `fn(&FieldType) -> bool` signature
185///
186/// This attribute works similarly to `skip_serializing_if` in `serde` â€“ if the function it points
187/// to returns `true` for the field value, the field will not be encoded as a label.
188///
189/// `Option` fields are skipped by default if they are `None` (i.e., they use `skip = Option::is_none`).
190///
191/// [`EncodeLabelSet`]: crate::traits::EncodeLabelSet
192///
193/// ## `unit`
194///
195/// **Type:** expression evaluating to [`Unit`]
196///
197/// Specifies unit of measurement for a label. The unit will be added to the label name as a suffix
198/// (e.g., `timeout_seconds` if placed on a field named `timeout`). This is mostly useful for [`Info`] metrics.
199///
200/// # Examples
201///
202/// ## Set with a single label
203///
204/// ```
205/// use derive_more::Display;
206/// use vise::{EncodeLabelSet, EncodeLabelValue};
207///
208/// #[derive(Debug, Display, Clone, PartialEq, Eq, Hash)]
209/// #[derive(EncodeLabelValue, EncodeLabelSet)]
210/// #[metrics(label = "method")]
211/// struct Method(&'static str);
212/// ```
213///
214/// ## Set with multiple labels
215///
216/// ```
217/// # use vise::EncodeLabelSet;
218/// #[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, EncodeLabelSet)]
219/// struct Labels {
220///     /// Label that is skipped when empty.
221///     #[metrics(skip = str::is_empty)]
222///     name: &'static str,
223///     /// Numeric label.
224///     num: u8,
225/// }
226/// ```
227///
228/// ## Set for info metric
229///
230/// ```
231/// use vise::{EncodeLabelSet, DurationAsSecs, Unit};
232/// use std::time::Duration;
233///
234/// #[derive(Debug, EncodeLabelSet)]
235/// struct InfoLabels {
236///     /// Simple label.
237///     version: &'static str,
238///     /// Label with a unit.
239///     #[metrics(unit = Unit::Seconds)]
240///     request_timeout: DurationAsSecs,
241///     /// Another label with a unit.
242///     #[metrics(unit = Unit::Bytes)]
243///     buffer_capacity: u64,
244/// }
245///
246/// let labels = InfoLabels {
247///     version: "0.1.0",
248///     request_timeout: Duration::from_millis(100).into(),
249///     buffer_capacity: 1_024,
250/// };
251/// // will be exported as the following labels:
252/// // { version="0.1.0", request_timeout_seconds="0.1", buffer_capacity="1024" }
253/// ```
254pub use vise_macros::EncodeLabelSet;
255/// Derives the [`EncodeLabelValue`] trait for a type, which encodes a metric label value.
256///
257/// The type for which the trait is derived must not have generics (lifetimes, type params or const params).
258/// The macro can be configured using `#[metrics()]` attributes.
259///
260/// # Container attributes
261///
262/// ## `format`
263///
264/// **Type:** string
265///
266/// **Default value:** `{}`
267///
268/// Specifies the format for the value as used in the `format!` macro etc. when encoding it to
269/// a label value. For example, `{}` means using [`Display`](std::fmt::Display).
270///
271/// [`EncodeLabelValue`]: trait@prometheus_client::encoding::EncodeLabelValue
272///
273/// ## `rename_all`
274///
275/// **Type:** string; one of `lowercase`, `UPPERCASE`, `camelCase`, `snake_case`, `SCREAMING_SNAKE_CASE`,
276/// `kebab-case`, `SCREAMING-KEBAB-CASE`
277///
278/// Specifies how enum variant names should be transformed into label values. This attribute
279/// can only be placed on enums in which all variants don't have fields (aka C-style enums).
280/// Mutually exclusive with the `format` attribute.
281///
282/// Caveats:
283///
284/// - `rename_all` assumes that original variant names are in `PascalCase` (i.e., follow Rust naming conventions).
285/// - `rename_all` requires original variant names to consist of ASCII chars.
286/// - Each letter of capitalized acronyms (e.g., "HTTP" in `HTTPServer`) is treated as a separate word.
287///   E.g., `rename_all = "snake_case"` will rename `HTTPServer` to `h_t_t_p_server`.
288///   Note that [it is recommended][clippy-acronyms] to not capitalize acronyms (i.e., use `HttpServer`).
289/// - No spacing is inserted before numbers or other non-letter chars. E.g., `rename_all = "snake_case"`
290///   will rename `Status500` to `status500`, not to `status_500`.
291///
292/// # Variant attributes
293///
294/// ## `name`
295///
296/// **Type:** string
297///
298/// Specifies the name override for a particular enum variant when used with the `rename_all` attribute
299/// described above.
300///
301/// [clippy-acronyms]: https://rust-lang.github.io/rust-clippy/master/index.html#/upper_case_acronyms
302///
303/// # Examples
304///
305/// ## Default format
306///
307/// Label value using the default `Display` formatting; note that `Display` itself is derived.
308///
309/// ```
310/// use derive_more::Display;
311/// use vise::EncodeLabelValue;
312///
313/// #[derive(Debug, Display, EncodeLabelValue)]
314/// struct Method(&'static str);
315/// ```
316///
317/// ## Custom format
318///
319/// Label value using `Hex` formatting with `0` padding and `0x` prepended.
320///
321/// ```
322/// # use derive_more::LowerHex;
323/// # use vise::EncodeLabelValue;
324/// #[derive(Debug, LowerHex, EncodeLabelValue)]
325/// #[metrics(format = "0x{:02x}")]
326/// struct ResponseType(u8);
327/// ```
328///
329/// ## Using `rename_all` on C-style enum
330///
331/// ```
332/// # use derive_more::LowerHex;
333/// # use vise::EncodeLabelValue;
334/// #[derive(Debug, EncodeLabelValue)]
335/// #[metrics(rename_all = "snake_case")]
336/// enum Database {
337///     Postgres, // renamed to "postgres"
338///     MySql, // renamed to "my_sql"
339///     #[metrics(name = "rocksdb")] // explicitly overrides the name
340///     RocksDB,
341/// }
342/// ```
343pub use vise_macros::EncodeLabelValue;
344/// Derives the [`Metrics`](trait@Metrics) trait for a type.
345///
346/// This macro must be placed on a struct with named fields and with no generics (lifetimes, type params
347/// or const params). Each field will be registered as metric or a family of metrics.
348/// The macro can be configured using `#[metrics()]` attributes.
349///
350/// # Container attributes
351///
352/// ## `prefix`
353///
354/// **Type:** string
355///
356/// Specifies a common prefix for all metrics defined in the type. If specified, the prefix will
357/// be prepended together with a `_` separator to a field name to get the metric name. Note that
358/// the metric name may be additionally transformed depending on the unit and metric type.
359///
360/// # Field attributes
361///
362/// ## `buckets`
363///
364/// **Type:** expression evaluating to a type implementing `Into<`[`Buckets`]`>`
365///
366/// Specifies buckets for a [`Histogram`] or a [`Family`] of `Histogram`s. This attribute is mandatory
367/// for these metric types and will result in a compile-time error if used on counters / gauges.
368///
369/// ## `unit`
370///
371/// **Type:** expression evaluating to [`Unit`]
372///
373/// Specifies unit of measurement for a metric. Note that specifying a unit influences the metric naming.
374///
375/// ## `labels`
376///
377/// **Type:** expression evaluating to an array `[&'static str; _]`
378///
379/// Specifies label names for a [`LabeledFamily`]. This attribute is mandatory for [`LabeledFamily`]
380/// and will result in a compile-time error if used with other metric types. The number of label names
381/// must match the number of label values in the `LabeledFamily` (i.e., its type).
382///
383/// # Examples
384///
385/// See crate-level docs and other crate docs for the examples of usage.
386pub use vise_macros::Metrics;
